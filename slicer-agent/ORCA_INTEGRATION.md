# OrcaSlicer Integration Notes (Protocol and Execution)

You have already implemented the AI dialog inside OrcaSlicer. Next you need to:
1. Call the `slicer-agent` HTTP service to get `actions[]` JSON.
2. In OrcaSlicer, do **second-stage validation + preview/confirmation + execution/rollback**.

## 1) Send Request
`POST http://<host>:<port>/v1/plan`

Request body (`PlanRequest`):
- `request_id`: generated by OrcaSlicer (for logs/trace)
- `user_text`: the dialog input
- `context`: provide minimal required context (material/nozzle/current parameter snapshot, etc.)
- `client_version`: optional

## 2) Receive Response
Response (`PlanResponse`):
- `requires_confirmation`: MVP recommends always `true`
- `actions`: a whitelist DSL actions list; each item contains `action` + `params`
- `errors/warnings`: if the model or parameter validation fails, `actions` may be empty

## 3) OrcaSlicer Local Executor (required)
You should implement an executor inside OrcaSlicer:
- Whitelist: only allow `action` in the set defined by `slicer-agent/dsl.py`
- Parameter validation: parse `params` per `action` and check ranges/units
  (e.g., file path / url / output filename)
- State preconditions: e.g. disallow `slice_current`/`slice_all` when no models are loaded
- Execution strategy:
  - Preview mode: generate and show preview to the user first
  - Execution failure: stop or rollback at action granularity (depends on OrcaSlicer API capabilities)

## 4) Action Mapping Table (MVP)
Current whitelist `action` supported by this `slicer-agent`:
- `load`
- `load_from_net`
- `slice_current`
- `slice_all`

Field mapping to Orca's `ActionExecutor` JSON command format:
- our `action` -> Orca `action`
- our `params` -> Orca `params`
